# Analyze External Access Slowness

## Goal

定位并缓解局域网内朋友电脑进入游戏后远端玩家移动卡顿的问题，优先解决运行时移动显示不平滑，而不是首屏加载问题。

## Confirmed Facts

- 项目是 Node.js + WebSocket + Three.js 的浏览器 3D 局域网对战游戏。
- `README.md` 明确目标场景是同一局域网试玩，并提示 Three.js 通过 CDN 加载，首次打开需要可访问 CDN。
- `index.html` 使用 import map 从 `https://unpkg.com/three@0.185.0/...` 加载 Three.js 和 addons；这会影响首次加载，但用户确认本次主要问题不是首次加载。
- `game.js` 是约 60KB 的客户端主逻辑，本地静态资源包含多个 GLB 模型；常用模型体积约 12KB-240KB，另有未在当前主逻辑中引用的 `models/soldier.glb` 约 2.1MB。
- `server.js` 用 `fs.readFile` 为每次静态资源请求读取文件，没有缓存头、压缩、ETag 或长缓存策略；这主要影响资源加载，不是本次移动卡顿的首要原因。
- 客户端加入后约每 45ms 发送一次 `move` 消息。
- 服务端每 50ms 对每个有玩家的房间广播完整 `state` 快照，快照包含所有玩家、排行榜、计分板和倒计时信息。
- 当前同步逻辑没有按房间人数、连接质量或数据变化做降频、差量更新、压缩或背压控制。
- 用户确认现场是同一局域网内朋友电脑进入游戏后移动卡顿，不是主要卡在首次加载页面。
- 原 `game.js` 在收到 `state` 后直接用 `r.mesh.position.set(p.x, 0, p.z)` 更新远端玩家位置，没有客户端插值、缓冲或预测；远端移动显示会受服务端 50ms 快照节奏限制。

## Requirements

- 分析外机访问卡顿的可能根因，并按证据强弱排序。
- 明确本次核心问题属于运行时远端玩家移动显示卡顿。
- 用最小客户端改动缓解局域网移动卡顿，不默认重写 WebSocket 协议或服务端同步循环。
- 保持射击判定尽量使用最新网络快照位置，避免视觉插值改变 gameplay 判定来源。
- 保持死亡、复活、初次出现、断开移除行为正确。

## Acceptance Criteria

- [x] 输出至少 3 个有代码或配置证据支撑的可能原因。
- [x] 区分首屏加载、网络同步、渲染负载、部署网络环境四类因素。
- [x] 确认本次用户现场主要是局域网运行时移动卡顿。
- [x] 远端玩家显示改为朝最新快照位置和朝向平滑插值。
- [x] 远端玩家初次出现、死亡、复活、大幅位移时直接贴近快照，避免拖影。
- [x] 射击检测临时使用远端最新快照位置，并在检测后恢复插值显示位置。
- [x] `node --check game.js` 和 `node --check server.js` 通过。
- [x] 本地 HTTP 首页探测返回 200。

## Out Of Scope

- 不在本任务中重写同步协议、做差量同步、压缩或服务端权威射线判定。
- 不引入打包器、新 CDN 或新后端框架。
- 不解决所有低配机器渲染掉帧问题。

## Follow-up Options

- 将服务端 `state` 拆分为高频移动包和低频计分/排行榜包。
- 增加服务端权威射线复核，避免客户端命中判定被篡改。
- 将 Three.js CDN 依赖本地化，改善首次加载稳定性。
