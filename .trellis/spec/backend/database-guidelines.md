# Database Guidelines

## Current Architecture

This project has no database, ORM, migration system, or persistent storage. Runtime data lives only in memory in `server.js`:

- `rooms` is a process-level `Map` keyed by `mode:mapId`.
- Each room owns `players` and `sockets` maps keyed by player ID.
- Scores, health, positions, round state, and timers are discarded when the process exits.

Do not introduce database terminology or persistence abstractions for ordinary gameplay changes. Adding persistence is an architectural change and must be planned explicitly.

## In-Memory Data Patterns

- Create rooms through `getOrCreateRoom(mode, mapId)` rather than writing to `rooms` at call sites.
- Build fresh room state with `createRoom`; keep room-owned collections inside the room object.
- Remove disconnected players from both `room.players` and `room.sockets` in the WebSocket `close` handler.
- Delete an empty room with the same key generated by `roomKey`.
- Derive snapshots and scoreboards from current room state; do not maintain duplicate cached rankings.

## Lifecycle and Consistency

Timers are part of in-memory state. `server.js` clears `player.respawnTimer` when a player disconnects and uses `deathId` to reject stale respawn callbacks. New delayed work must follow the same pattern: retain enough identity to detect stale callbacks and clean timers during owner teardown.

## Non-Applicable Areas

There are currently no tables, transactions, query naming rules, migrations, or seed scripts. If a task adds a database, update this guide in the same task with the selected storage technology, lifecycle, and migration commands.
